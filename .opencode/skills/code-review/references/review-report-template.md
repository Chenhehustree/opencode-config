# 代码审查报告模板

生成代码审查报告时使用此模板。根据 Step 1（PR 信息）和 Step 2（代码审查）的产出填充各节内容。

---

以下是模板内容：

```markdown
# Code Review: PR #<编号> - <PR 标题>

## PR 概要

- **标题**: <PR 标题>
- **变更文件数**: <N> 个文件
- **关联 Plan**: `<plan 路径>`
- **任务数**: <N> 个任务，<已完成数> 个已完成

## 审查结果: **<需修复 | 审查通过>**

---

## 发现的问题

<!-- 按严重级别分组列出所有问题。若该级别无问题则省略对应小节。 -->

### Blocker 级别问题 (<N> 个)

<!-- 必须修复才能合并：正确性错误、安全问题、关键测试缺失 -->

#### 1. `<文件路径>` L<行号>: <问题简述>

- **问题**: <详细描述>
- **建议**: <具体修复建议>

#### 2. ...

### Warning 级别问题 (<N> 个)

<!-- 建议修复但不阻塞合并：代码组织、命名不一致、业务目标不清晰 -->

#### 1. `<文件路径>` L<行号>: <问题简述>

- **问题**: <详细描述>
- **建议**: <具体修复建议>

### Suggestion 级别问题 (<N> 个)

<!-- 可选的改进建议：性能优化、可读性提升 -->

#### 1. `<文件路径>` L<行号>: <问题简述>

- **问题**: <详细描述>
- **建议**: <具体修复建议>

---

## 质量检查结果

<!-- 列出 Step 2 中执行的各项质量检查及其结果 -->

| 检查项 | 结果 |
|--------|------|
| 测试 | <结果，如：N 个测试全部通过 ✅ 或 M 个失败 ❌> |
| 语法检查 | <结果> |
| 安全 | <结果> |
| 目录结构 | <结果> |

---

## 总结

| 严重级别 | 数量 |
|---------|------|
| **blocker** | <N> |
| **warning** | <N> |
| **suggestion** | <N> |

<!-- 若需修复：说明主要问题集中在哪里、影响是什么、修复建议 -->
<!-- 若审查通过：简要总结代码质量和亮点 -->

**<需要修复的问题 | 审查通过>**: <一段总结性文字，概括主要发现和建议>

**建议**: <下一步行动建议>
```
